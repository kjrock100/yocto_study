# 시스템 요구 사항

Yocto 프로젝트 참조 매뉴얼에 오신 것을 환영합니다. 이 매뉴얼은 현재 Yocto 프로젝트 릴리스에 대한 참조 정보를 제공하며, Yocto 프로젝트의 기본을 이해한 후에 가장 효과적으로 사용됩니다. 이 매뉴얼은 Yocto 프로젝트의 시작점으로 읽거나 처음부터 끝까지 읽을 의도로 작성되지 않았습니다. 대신, Yocto 프로젝트를 사용하는 동안 필요에 따라 변수 정의, 클래스 설명 등을 찾기 위해 이 매뉴얼을 사용하세요.

Yocto 프로젝트에 대한 입문 정보는 [Yocto 프로젝트 웹사이트](https://www.yoctoproject.org/)와 Yocto 프로젝트 개요 및 개념 매뉴얼의 "Yocto 프로젝트 개발 환경" 장을 참조하세요.

Yocto 프로젝트를 사용하여 개념을 이해할 필요 없이 빠르게 이미지를 빌드하고 싶다면, [Yocto 프로젝트 빠른 빌드](https://docs.yoctoproject.org/brief-yoctoprojectqs/index.html) 문서를 참조하세요. "방법" 정보는 [Yocto 프로젝트 개발 작업 매뉴얼](https://docs.yoctoproject.org/dev-manual/index.html)에서 찾을 수 있습니다. Yocto 프로젝트 개요 및 개념 정보는 [Yocto 프로젝트 개요 및 개념 매뉴얼](https://docs.yoctoproject.org/overview-manual/index.html)에서 찾을 수 있습니다.

## 1.1 최소 여유 디스크 공간

`qemux86-64` 머신에 대해 `core-image-sato`와 같은 이미지를 빌드하려면 최소 90GB의 여유 디스크 공간이 있는 시스템이 필요합니다. 그러나 더 복잡한 이미지를 빌드하고, 여러 빌드를 실행하고, 빌드 아티팩트를 캐시하여 빌드 효율성을 개선하기 위해서는 훨씬 더 많은 디스크 공간이 필요합니다.

디스크 공간이 부족한 경우, 개발 작업 매뉴얼의 "[디스크 공간 절약](https://docs.yoctoproject.org/dev-manual/disk-space.html)" 섹션을 참조하세요.

## 1.2 최소 시스템 RAM

`qemux86-64` 머신에 대해 `core-image-sato`와 같은 이미지를 4개의 CPU 코어가 있는 오래된 시스템에 최소 32GB의 RAM으로 빌드할 수 있습니다. 그러나 가능한 한 많은 RAM과 많은 CPU 코어가 있는 시스템에서 빌드가 훨씬 더 빠를 것입니다.

- Fedora 39
- Fedora 40
- OpenSUSE Leap 15.5
- Ubuntu 20.04 (LTS)
- Ubuntu 24.10

Yocto 프로젝트는 이러한 버전 중 일부가 가질 수 있는 비공개 업데이트에 접근할 수 없습니다. 따라서 이러한 업데이트에 접근할 수 있는 경우 테스트의 가치가 제한적입니다. 또한 Python과 같은 충분히 최신 도구를 얻기 위해 구형 릴리스에서 [buildtools](https://docs.yoctoproject.org/ref-manual/terms.html#term-buildtools)를 사용해야 할 수 있습니다.

마지막으로, "Walnascar"의 이전 수정본에서 이전에 테스트되었지만 더 이상 테스트되지 않는 배포판 버전은 다음과 같습니다:

- Ubuntu 23.04

> **참고:**
>
> - Yocto 프로젝트 팀은 모든 Yocto 프로젝트 릴리스가 공식적으로 지원되는 모든 Linux 배포판과 100% 호환 가능하도록 하려고 노력하지만, 특정 배포판에서만 발생하는 문제가 발생할 수 있습니다.
>
> - Yocto 프로젝트 릴리스는 위의 목록에 있는 안정적인 Linux 배포판에 대해 테스트됩니다. Yocto 프로젝트는 다른 배포판에서도 작동해야 하지만 이에 대해서는 검증이 수행되지 않습니다.
>
> - 특히, Yocto 프로젝트는 롤링 릴리스 또는 개발 배포판을 지원하지 않으며 현재 이들을 지원할 계획이 없습니다(지속적으로 변화하는 성질 때문). 패치 및 버그 보고를 환영하지만 우리의 우선순위는 위에 나열된 지원되는 플랫폼임을 명심하세요.
>
> - Linux 배포판이 위의 목록에 없는 경우, 일반적으로 `scripts/install-buildtools`를 실행하여 Yocto 프로젝트 릴리스에 필요한 호스트 도구를 포함하는 [buildtools](https://docs.yoctoproject.org/ref-manual/terms.html#term-buildtools) 또는 [buildtools-extended](https://docs.yoctoproject.org/ref-manual/terms.html#term-buildtools-extended) tarball을 얻을 것을 권장합니다.
>
> - Windows 10 이상 또는 Windows Server 2019 이상에서 Windows Subsystem For Linux v2를 사용하여 빌드 호스트를 설정할 수 있지만, WSL 2를 사용하는 빌드 호스트에 대해서는 검증이 수행되지 않습니다.
>
> - 문제가 발생하면 [Yocto 프로젝트 Bugzilla](https://bugzilla.yoctoproject.org/)로 이동하여 버그를 제출하세요. 우리는 당신의 경험에 대해 알고 싶습니다. 버그 제출 방법에 대한 정보는 Yocto 프로젝트 [Bugzilla wiki 페이지](https://wiki.yoctoproject.org/wiki/Bugzilla_Configuration_and_Bug_Tracking)와 Yocto 프로젝트 및 OpenEmbedded 기여자 가이드의 "[Yocto 프로젝트 및 OpenEmbedded에 대한 결함 보고](https://docs.yoctoproject.org/contributor-guide/report-defect.html)" 섹션을 참조하세요.

## 1.4 빌드 호스트에 필요한 패키지

호스트 개발 시스템에 필요한 패키지 목록은 Yocto 프로젝트를 사용한 모든 빌드 시나리오를 다룰 때 클 수 있습니다. 이 섹션에서는 Linux 배포판 및 기능에 따라 필요한 패키지를 설명합니다.

### 1.4.1 Ubuntu 및 Debian

다음은 지원되는 Ubuntu 또는 Debian Linux 배포판으로 헤드리스 시스템에서 이미지를 빌드하는 데 필요한 패키지입니다:

```bash
sudo apt-get install build-essential chrpath cpio debianutils diffstat file gawk gcc git iputils-ping libacl1 liblz4-tool locales python3 python3-git python3-jinja2 python3-pexpect python3-pip python3-subunit socat texinfo unzip wget xz-utils zstd
```

또한 `en_US.UTF-8` 로케일이 활성화되어 있는지 확인해야 합니다:

```bash
$ locale --all-locales | grep en_US.utf8
```

그렇지 않은 경우, `locales` 패키지를 재구성하여 추가할 수 있습니다(대화형 셸 필요):

```bash
$ sudo dpkg-reconfigure locales
```

> **참고:**
>
> - 대화형 셸에 없는 경우 `dpkg-reconfigure`이 예상대로 작동하지 않습니다. 로케일을 추가하려면 `/etc/locale.gen` 파일을 편집하여 `en_US.UTF-8` 로케일을 추가/주석 해제해야 합니다. 루트로 이것을 하는 간단한 방법은:
>
> ```bash
> $ echo "en_US.UTF-8 UTF-8" >> /etc/locale.gen
> $ locale-gen
> ```
>
> - 빌드 시스템에 `oss4-dev` 패키지가 설치되어 있는 경우, 패키지가 Debian 시스템에 자신의 커스텀 `/usr/include/linux/soundcard.h`를 설치하기 때문에 QEMU 빌드 실패가 발생할 수 있습니다. 이 상황에 빠지면 다음 중 하나를 시도해보세요:
>
> ```bash
> $ sudo apt build-dep qemu
> $ sudo apt remove oss4-dev
> ```

프로젝트 설명서 매뉴얼을 빌드하는 데 필요한 패키지는 다음과 같습니다:

```bash
sudo apt install git librsvg2-bin locales make python3-saneyaml python3-sphinx-rtd-theme sphinx
```

위의 패키지에 추가로, PDF 형식의 문서를 빌드하는 데 필요한 패키지는 다음과 같습니다:

```bash
sudo apt install fonts-freefont-otf latexmk tex-gyre texlive-fonts-extra texlive-fonts-recommended texlive-lang-all texlive-latex-extra texlive-latex-recommended texlive-xetex
```

### 1.4.4 AlmaLinux 패키지

다음은 지원되는 AlmaLinux 배포판으로 헤드리스 시스템에서 이미지를 빌드하는 데 필요한 패키지입니다:

```bash
sudo dnf install -y epel-release
```

>


> **경고:**
>
> Fedora 또는 OpenSUSE와 달리 AlmaLinux는 `texlive-collection-fontsextra`, `texlive-collection-lang*` 및 `texlive-collection-latexextra` 패키지를 제공하지 않으므로 문제가 발생할 수 있습니다. 이는 [tlmgr](https://tug.org/texlive/tlmgr.html)을 사용하여 설치할 수 있습니다.

위의 패키지에 추가로, PDF 형식의 문서를 빌드하는 데 필요한 패키지는 다음과 같습니다:

```bash

```

## 1.5 필수 Git, tar, Python, make 및 gcc 버전

빌드 시스템을 사용하려면 호스트 개발 시스템이 Git, tar 및 Python에 대해 다음 버전 요구 사항을 충족해야 합니다:

- Git 1.8.3.1 이상
- tar 1.28 이상
- Python 3.9.0 이상
- GNU make 4.0 이상

호스트 개발 시스템이 이 모든 요구 사항을 충족하지 않는 경우, 이러한 도구를 포함하는 [buildtools](https://docs.yoctoproject.org/ref-manual/terms.html#term-buildtools) tarball을 설치하여 이를 해결할 수 있습니다. 미리 빌드된 tarball을 다운로드하거나 BitBake를 사용하여 생성할 수 있습니다.

또한 호스트 개발 시스템은 gcc에 대해 다음 버전 요구 사항을 충족해야 합니다:

- gcc 10.1 이상

호스트 개발 시스템이 이 요구 사항을 충족하지 않는 경우, 추가 도구(Debian/Ubuntu `build-essential` 패키지와 동등)를 포함하는 [buildtools-extended](https://docs.yoctoproject.org/ref-manual/terms.html#term-buildtools-extended) tarball을 설치하여 이를 해결할 수 있습니다.

손상된 make 버전이 있는 시스템(예: 패치 없는 make 4.2.1)이지만 나머지 호스트 도구는 사용 가능한 경우, 대신 [buildtools-make](https://docs.yoctoproject.org/ref-manual/terms.html#term-buildtools-make) tarball을 사용할 수 있습니다.

다음 섹션에서는 [buildtools](https://docs.yoctoproject.org/ref-manual/terms.html#term-buildtools), [buildtools-extended](https://docs.yoctoproject.org/ref-manual/terms.html#term-buildtools-extended) 또는 [buildtools-make](https://docs.yoctoproject.org/ref-manual/terms.html#term-buildtools-make) 도구 집합을 설치하는 세 가지 방법을 설명합니다.

### 1.5.1 `install-buildtools` 스크립트를 사용하여 미리 빌드된 `buildtools` Tarball 설치

`install-buildtools` 스크립트는 이러한 도구를 얻을 수 있는 세 가지 방법 중 가장 쉬운 방법입니다. 미리 빌드된 [buildtools](https://docs.yoctoproject.org/ref-manual/terms.html#term-buildtools) 설치 프로그램을 다운로드하고 자동으로 도구를 설치합니다:

1. **`install-buildtools` 스크립트를 실행합니다. 다음은 예제입니다:**

```bash
$ cd poky
$ scripts/install-buildtools \
  --without-extended-buildtools \
  --base-url https://downloads.yoctoproject.org/releases/yocto \
  --release yocto-5.2.999 \
  --installer-version 5.2.999
```

실행 중에 [buildtools](https://docs.yoctoproject.org/ref-manual/terms.html#term-buildtools) tarball이 다운로드되고, 다운로드의 체크섬이 검증되고, 설치 프로그램이 실행되고, 설치가 작동하는지 확인하기 위해 일부 기본 검사가 실행됩니다.

`sudo` 권한의 필요성을 피하기 위해 `install-buildtools` 스크립트는 기본적으로 설치 프로그램에게 다음 경로에 설치하도록 지시합니다:

```
/path/to/poky/buildtools
```

호스트 개발 시스템이 [buildtools-extended](https://docs.yoctoproject.org/ref-manual/terms.html#term-buildtools-extended) tarball에서 제공하는 추가 도구가 필요한 경우, 기본 매개변수로 `install-buildtools` 스크립트를 대신 실행할 수 있습니다:

```bash
$ cd poky
$ scripts/install-buildtools
```

또는 호스트 개발 시스템에 손상된 `make` 버전이 있는 경우(알려진 좋은 버전의 `make`만 필요), `--make-only` 옵션을 사용할 수 있습니다:

```bash
$ cd poky
$ scripts/install-buildtools --make-only
```

2. **다음과 같은 명령을 사용하여 도구 환경 설정 스크립트를 소싱합니다:**

```bash
$ source /path/to/poky/buildtools/environment-setup-x86_64-pokysdk-linux
```

설정 스크립트를 소싱한 후, 도구가 `PATH`에 추가되고 도구를 실행하는 데 필요한 다른 환경 변수가 초기화됩니다.

그 결과는 Git, tar, Python 및 `chrpath`의 작동 버전입니다. [buildtools-extended](https://docs.yoctoproject.org/ref-manual/terms.html#term-buildtools-extended) tarball의 경우 `packagegroup-core-buildessential`에 포함된 다른 도구를 포함하여 `gcc` 및 `make` 같은 도구의 추가 작동 버전입니다.

3. **설치 스크립트가 지시하는 대로 도구 환경 설정 스크립트를 소싱합니다:**

### 1.5.3 자신의 `buildtools` Tarball 빌드

자신의 [buildtools](https://docs.yoctoproject.org/ref-manual/terms.html#term-buildtools) 설치 프로그램을 빌드하고 실행하는 것은 BitBake를 이미 실행할 수 있는 빌드 호스트가 있을 때만 적용됩니다. 이 경우 해당 머신을 사용하여 `.sh` 파일을 빌드한 다음 최소 Git, tar 및 Python(또는 gcc) 요구 사항을 충족하지 않는 머신으로 전송하고 실행하기 위한 단계를 취합니다.

자신의 [buildtools](https://docs.yoctoproject.org/ref-manual/terms.html#term-buildtools) 설치 프로그램을 빌드하고 실행하기 위한 단계는 다음과 같습니다:

1. **BitBake를 실행할 수 있는 머신에서 설정 스크립트([oe-init-build-env](https://docs.yoctoproject.org/ref-manual/structure.html#structure-core-script))를 사용하여 빌드 환경을 설정했는지 확인합니다.**

2. **Tarball을 빌드하기 위해 BitBake 명령을 실행합니다:**

```bash
$ bitbake buildtools-tarball
```

또는 확장 tarball을 빌드하려면:

```bash
$ bitbake buildtools-extended-tarball
```

또는 make 전용 tarball을 빌드하려면:

```bash
$ bitbake buildtools-make-tarball
```

> **참고:** `local.conf` 파일의 [SDKMACHINE](https://docs.yoctoproject.org/ref-manual/variables.html#term-SDKMACHINE) 변수는 32비트 또는 64비트 시스템용 도구를 빌드할지 여부를 결정합니다.

빌드가 완료되면 도구를 설치하는 `.sh` 파일을 [빌드 디렉토리](https://docs.yoctoproject.org/ref-manual/terms.html#term-Build-Directory)의 `tmp/deploy/sdk` 하위 디렉토리에서 찾을 수 있습니다. 설치 프로그램 파일에는 이름에 "buildtools" 또는 "buildtools-extended"이 있습니다.

3. **빌드 호스트에서 `.sh` 파일을 Git, tar 또는 Python(또는 gcc) 요구 사항을 충족하지 않는 머신으로 전송합니다.**

4. **이 머신에서 `.sh` 파일을 실행하여 도구를 설치합니다. 다음은 전통적인 설치 프로그램의 예제입니다:**

```bash
$ sh ~/Downloads/x86_64-buildtools-nativesdk-standalone-5.2.999.sh
```

확장 설치 프로그램의 경우:

```bash
$ sh ~/Downloads/x86_64-buildtools-extended-nativesdk-standalone-5.2.999.sh
```

make 전용 설치 프로그램의 경우:

```bash
$ sh ~/Downloads/x86_64-buildtools-make-nativesdk-standalone-5.2.999.sh
```

실행 중에 설치 디렉토리를 선택할 수 있는 프롬프트가 나타납니다. 예를 들어 다음을 선택할 수 있습니다: `/home/your_username/buildtools`

5. **다음과 같은 명령을 사용하여 도구 환경 설정 스크립트를 소싱합니다:**

```bash
$ source /home/your_username/buildtools/environment-setup-x86_64-poky-linux
```

설정 스크립트를 소싱한 후, 도구가 `PATH`에 추가되고 도구를 실행하는 데 필요한 다른 환경 변수가 초기화됩니다.

그 결과는 Git, tar, Python 및 `chrpath`의 작동 버전입니다. [buildtools-extended](https://docs.yoctoproject.org/ref-manual/terms.html#term-buildtools-extended) tarball의 경우 `packagegroup-core-buildessential`에 포함된 다른 도구를 포함하여 `gcc` 및 `make` 같은 도구의 추가 작동 버전입니다.
