# Yocto Project 소스 (Sources)

Yocto Project 빌드 시스템은 패키지를 생성하기 위해 다양한 위치에서 **소스 코드(Sources)**를 가져옵니다.

## 1. 개요

BitBake는 레시피 파일(`.bb`)에 정의된 `SRC_URI` 변수를 사용하여 소스 코드의 위치를 파악하고 다운로드(Fetch)합니다. 다운로드된 소스는 `DL_DIR` 디렉토리에 저장되어 재사용됩니다.

## 2. 소스의 종류

### 2.1 업스트림 프로젝트 릴리스 (Upstream Project Releases)
대부분의 오픈 소스 프로젝트는 특정 버전의 소스 코드를 압축 파일(Tarball) 형태로 배포합니다.
*   **형식:** `.tar.gz`, `.tar.bz2`, `.zip` 등.
*   **특징:** 안정적인 릴리스 버전을 사용할 때 주로 사용됩니다.
*   **예시:** `SRC_URI = "https://example.com/downloads/app-1.0.tar.gz"`

### 2.2 로컬 프로젝트 (Local Projects)
소스 코드가 Yocto 레이어 내부에 직접 포함되어 있거나, 로컬 파일 시스템에 존재하는 경우입니다.
*   **용도:** 설정 파일(`defconfig`), 패치 파일(`.patch`), 또는 작은 규모의 커스텀 애플리케이션 소스.
*   **예시:** `SRC_URI = "file://my-config.conf"`

### 2.3 소스 제어 관리자 (Source Control Managers - SCM)
Git, Subversion(SVN) 등의 버전 관리 시스템에서 직접 소스를 가져옵니다.
*   **장점:** 특정 커밋 해시, 태그, 브랜치를 지정하여 빌드할 수 있어 개발 중인 최신 코드를 추적하기 용이합니다.
*   **예시:** `SRC_URI = "git://github.com/example/app.git;protocol=https;branch=main"`

## 3. 소스 미러 (Source Mirrors)

네트워크 불안정이나 업스트림 서버의 다운타임에 대비하기 위해 Yocto Project는 미러링(Mirroring) 기능을 제공합니다.

### 3.1 PREMIRRORS
업스트림 소스를 시도하기 **전**에 먼저 확인하는 미러 사이트입니다. 주로 자체적으로 구축한 소스 서버를 지정하여 빌드 속도를 높이고 외부 네트워크 의존성을 줄이는 데 사용됩니다.

### 3.2 MIRRORS
업스트림 소스 다운로드가 **실패했을 때** 확인하는 대체 사이트입니다. Yocto Project는 주요 오픈 소스 패키지들에 대한 공용 미러를 제공합니다.

## 4. 다운로드 디렉토리 (DL_DIR)

모든 다운로드된 소스 파일(tarball)과 SCM 저장소의 복사본은 `local.conf`에 정의된 `DL_DIR` 경로에 저장됩니다.
*   이 디렉토리를 여러 빌드 프로젝트가 공유하도록 설정하면, 중복 다운로드를 방지하여 빌드 시간을 크게 단축할 수 있습니다.

---
*참고: `SRC_URI`에는 소스 위치뿐만 아니라 적용할 패치 파일이나 설정 파일도 함께 나열됩니다.*
